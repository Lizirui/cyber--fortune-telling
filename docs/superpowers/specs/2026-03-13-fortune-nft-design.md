# Fortune NFT - 赛博算命 NFT 设计文档

## 1. 项目概述

**项目名称**: Fortune NFT (赛博算命 / Cyber Fortune)

**项目定位**: 一个趣味盲盒 NFT 项目，用户 mint 后可获得由 AI 实时生成的 60-80 字祝福语，稀有度在 mint 后揭晓，为用户带来"开盲盒"的惊喜体验。

**目标用户**: Web3 爱好者、NFT 收藏者、追求有趣数字纪念品的用户

## 2. 功能规格

### 2.1 Mint 盲盒

- 用户连接钱包后，支付固定 mint 费用获得一个 NFT
- **Mint 费用**: $0.01 BASE (或等值代币)
- NFT 内容为 AI 实时生成的 60-80 字中文祝福语
- 稀有度在铸造时由后端服务随机判定，用户可在 NFT 详情页查看
- 每个 NFT 的祝福语都是独一无二的

### 2.2 稀有度系统

| 稀有度 | 概率  | 描述           |
| ------ | ----- | -------------- |
| N      | 60%   | 基础祝福语     |
| R      | 28.7% | 较好的祝福语   |
| SR     | 10%   | 精美的祝福语   |
| SSR    | 1%    | 极好的祝福语   |
| SP     | 0.25% | 绝佳的祝福语   |
| UR     | 0.05% | 梦幻极光祝福语 |

稀有度在 mint 时由后端服务随机判定，NFT 上链时一并写入合约。

### 2.3 交易市场

- 用户可以将拥有的 NFT 挂单出售，设定价格
- 其他用户可以购买挂单中的 NFT
- 支持在项目网站内完成所有交易流程

### 2.4 OpenSea 兼容

- NFT 遵循 ERC-721 标准
- 可直接在 OpenSea 等第三方市场交易

## 3. 技术架构

### 3.1 区块链层

- **主链**: Base (EVM L2)
- **标准**: ERC-721
- **链上存储**: 祝福语文本 + 稀有度等级

### 3.2 智能合约

**NFT 合约功能**:

- `mint()` - 用户铸造 NFT，支付固定费用
- `listItem()` - 挂单出售
- `buyItem()` - 购买NFT
- `cancelListing()` - 取消挂单
- `tokenURI()` - 返回 NFT 元数据（含祝福语和稀有度）

### 3.3 后端服务

**AI 生成服务**:

- 接收 mint 请求
- 调用 AI API (DeepSeek) 生成 60-80 字祝福语
- 随机判定稀有度
- 返回祝福语 + 稀有度给前端
- **成本**: 约 $0.001/次

**祝福语 Prompt 模板**:

```
请生成一个60-80字的中文祝福语，要求：
1. 积极向上，正能量
2. 可以包含一些玄学/命运相关的元素
3. 语言优美，有诗意
4. 不要使用特殊符号和表情
```

### 3.4 前端 DApp

**页面结构**:

- 首页 - Mint 盲盒入口、排行榜入口
- Mint 页面 - 点击 mint、揭晓动画、查看结果
- 市场页面 - 浏览/搜索在售 NFT、购买
- 排行榜页面 - 欧皇榜、Mint 次数榜
- 个人中心 - 查看拥有的 NFT、管理挂单

**核心交互**:

- 钱包连接 (RainbowKit/Wagmi)
- Mint 流程：连接钱包 → 点击 Mint → 等待 AI 生成 → 揭晓动画 → 显示祝福语和稀有度
- 交易流程：选择 NFT → 设定价格挂单 / 购买 → 确认交易

## 4. 数据流

```
用户点击 Mint
    ↓
前端调用后端 API 请求生成祝福语
    ↓
后端调用 AI API 生成祝福语 + 随机判定稀有度
    ↓
后端用私钥签名 (祝福语 + 稀有度 + nonce)
    ↓
后端返回 (祝福语 + 稀有度 + 签名) 给前端
    ↓
前端调用合约 mint(祝福语, 稀有度, 签名)
    ↓
合约验证签名，确认数据未被篡改
    ↓
交易确认，NFT 铸造成功
    ↓
前端展示祝福语和稀有度
```

**签名验证机制**: 后端用项目私钥对 (祝福语, 稀有度, 时间戳, 过期时间) 进行签名，合约在 mint 时验证：

1. 签名来自授权的后端地址
2. 签名未过期（block.timestamp < expiresAt）
3. 该签名未被使用过（防止重放攻击）

## 5. 关键设计决策

1. **链上存储祝福语**: 保证 NFT 数据的去中心化和持久性，Base Gas 足够低
2. **后端判定稀有度 + 签名验证**: 避免链上随机数被预测，防止用户篡改数据
3. **签名验证机制**: 合约验证后端签名，确保祝福语和稀有度真实
4. **链上市场 + OpenSea 兼容**: 自建市场保证交易可验证，OpenSea 扩大流动性

## 6. 扩展设计

### 6.1 供应量限制

- **最大供应量**: 10,000 个 NFT
- **每个钱包限 Mint**: 每次交易最多 Mint 5 个，每天最多 10 个（防止机器人批量 Mint）

### 6.2 版税机制

- **创作者版税**: 每次二次销售收取 5% 的费用
- 使用 ERC-2981 标准实现（NFT Royalty Standard）

### 6.3 元数据策略

- **祝福语存储**: 链上存储（Base Gas 足够低）
- **NFT 图片**: 后端动态生成 SVG，根据稀有度使用不同模板（边框颜色、背景装饰、发光效果等），转为 base64 data URI
- **稀有度模板**:
  - N: 灰色边框，深色背景
  - R: 蓝色边框，轻微发光
  - SR: 紫色边框，星星装饰
  - SSR: 金色边框，金光闪烁效果
  - SP: 红色边框，强烈红光动画（概率 1%）
  - UR: 梦幻极光渐变（粉紫→天蓝→米白），极光流动效果，边缘发光
- **OpenSea 兼容**: 实现标准 tokenURI，返回 JSON 元数据包含:
  - `image`: base64 SVG data URI（格式: `data:image/svg+xml;base64,...`）
  - `attributes`: 稀有度属性
  - `description`: 祝福语文本

### 6.5 排行榜系统

- **欧皇榜**: 按用户拥有高稀有度 NFT 数量排序（稀有度权重: UR > SP > SSR > SR > R > N）
- **Mint 次数榜**: 按用户总 mint 次数排序
- **展示**: 显示钱包地址（格式: 0x1234...abcd）
- **数据源**: 从合约事件中索引，存储在后端数据库

### 6.6 交易市场架构

- **链上市场**: 使用合约实现挂单/购买逻辑（确保交易可验证）
- **挂单信息**: 存储在合约中（seller, price, tokenId）
- **购买流程**: 买家支付 ETH/Base 币购买，款项直接转给卖家
- **OpenSea 集成**: NFT 可在 OpenSea 挂单交易（通过 OpenSea 的 Seaport 协议），与链上市场互不影响
- **支付代币**: 支持 ETH (Base 币)
